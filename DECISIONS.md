| 28 | 2026-04-26 | scheduling  | Adopt SemaClaw §3.6's four-mode scheduled-task taxonomy: `notification`, `script`, `synthesis`, `synthesisNotify`. Replaces the two-mode (`deterministic`, `synthesis`) model from CLAUDE.md §8. Stored as a string in the existing `reminders.mode` text column — no schema migration. **`notification`** (zero-token, user-visible) — AlarmManager + flutter_local_notifications fires a system notification rendered from a stored template. The common case: flea, heartworm, vaccine, weight-check reminders. **`script`** (zero-token, no notification) — WorkManager runs a registered Dart task at fire time. Side effects only — write a journal entry, refresh embeddings, vacuum stale FTS rows. **`synthesis`** (LLM call, no notification) — WorkManager runs a scheduled background turn that writes a markdown entry under `wiki/<id>/digest/`. The existing Phase 3.7 weekly-summary runner is the canonical instance. **`synthesisNotify`** (LLM call + notification) — synthesis then a "Loki's weekly summary is ready" notification. Stubbed in Phase 4 with `UnimplementedError('Phase 5 — Pro-tier weekly summary notifications')` so the dispatcher's switch-on-mode is exhaustive; concrete consumer ships in Phase 5 alongside Pro-tier billing. **Why four, not three or five.** Two binary axes — token-burning × notification-firing — make four cells. Folding `synthesisNotify` into a `payload.notify=true` boolean obscures the cost dimension at the dispatcher layer (notification permission has to gate that cell specifically). Splitting `script` further into "battery-aware" vs "exact-time" is over-engineering: WorkManager handles condition-gated work; AlarmManager handles exact-time. **Default mode for the agent's `schedule_reminder` tool.** When the LLM calls `schedule_reminder` without specifying, mode defaults to `notification` — the overwhelmingly common case. The tool schema validates `mode ∈ {notification, script, synthesis, synthesisNotify}` and rejects unknown values rather than silently downgrading. **Reminders route placement.** Phase 4 lands a top-level `/reminders` route; the free tier is single-pet so there's no ambiguity. Phase 5 multi-pet may re-parent it under per-pet detail (parallel to how a future Profile-per-pet sub-tree will work) — flagged here so future Claude knows this isn't a permanent placement. | —          |
| 29 | 2026-04-26 | agent       | Phase 4 medical-safety pre-screener — design lock. **Liability framing.** Highest legal-exposure surface in the app. PRODUCT.md "What PetPal is NOT" already commits the escalation runs as deterministic code, not as a prompt instruction, so a model failure cannot hide an emergency. **Tradeoff direction.** Bias toward false positives. A user nudged to call the vet when not strictly urgent loses thirty seconds of attention; a missed real emergency could lose the pet. The asymmetry isn't close. Concretely: matches are case-insensitive, regexes use word boundaries but not strict negation lookaheads, partial matches in multi-symptom AND groups count, and the screener never *prevents* the LLM from running — it only prepends a directive to the system prompt and surfaces a UI badge. **Defense in depth.** Three layers: (1) **code primary** — the pattern table at `lib/harness/guardrails/red_flags.dart` covering eleven categories (`blood_in_stool`, `blood_in_vomit`, `repeat_vomit`, `seizure`, `bloat`, `pale_gums`, `toxin_ingestion`, `dyspnea`, `collapse`, `trauma_fracture`, `lethargy_anorexia`); (2) **prompt backup** — the SessionBuilder Output-contract block instructs the model to recognise urgent symptoms the screener may have missed and open with the verbatim escalation preamble; (3) **user-visible audit** — the chat-screen badge surfaces *which* category fired so the user can scrollback-verify and we can ship coverage updates from real reports. **Coverage rule.** Every category ships with ≥30 positive phrasings + ≥20 negative phrasings in `test/harness/guardrails/red_flags_fixture.dart`. New patterns require new fixtures in the same commit. **Escalation copy.** Locked verbatim in VOICE.md §6 example 10: *"This sounds urgent — please call your vet or an emergency animal hospital now. PetPal is software, not a vet. I can help you write down what's happening so it's ready when you call."* The screener directive instructs the LLM to open with this exact text. **Input scope.** Phase 4 screens chat input only — never wiki-entry text the user types directly, since wiki entries are legitimately retrospective (a vet visit recorded after the fact may name urgent symptoms that are no longer urgent). Phase 6+ may extend to photo captions. **Badge styling.** Subdued in scrollback (small `Icons.warning_amber_rounded` icon + muted color) but prominent at the moment of escalation (the live preamble is the alert; the historical badge is a marker that this happened). The badge persists forever — it's a historical record, not a current-state indicator — and survives any future edit-message or mark-resolved (Phase 5+) feature. | —          |
